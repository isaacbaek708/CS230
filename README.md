# CS 230 Portfolio Reflection

## Briefly summarize The Gaming Room client and their software requirements.

The Gaming Room wanted to expand its existing game, *Draw It or Lose It*, into a web-based application that could support multiple teams and players across different operating systems. The client needed a design that was scalable, secure, and capable of serving multiple users through a client-server architecture while maintaining a consistent experience across platforms.

## What did you do particularly well in developing this documentation?

I think I did a good job organizing the software design document and explaining the technical recommendations in a way that both developers and non-technical stakeholders could understand. Breaking the design into sections made it easier to connect the client's requirements with the proposed software architecture and design decisions.

## What about the process of working through a design document did you find helpful when developing the code?

Creating the design document before writing code helped me understand the overall structure of the application. It encouraged me to think about system architecture, operating system support, security, and scalability before implementation. As I worked on the coding assignments later in the course, I had a much better understanding of how the different components fit together.

## If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I were to revise one section, I would expand the discussion of development tools and platform-specific technologies. After receiving instructor feedback earlier in the course, I realized that providing more detail about recommended programming languages, IDEs, and frameworks for Windows, macOS, and Linux would make the design document more complete and practical.

## How did you interpret the user's needs and implement them into your software design? Why is it so important to consider the user's needs when designing?

I focused on the client's primary goals, which were supporting multiple operating systems, maintaining security, and allowing the application to grow as more users joined. These needs influenced decisions such as using a client-server architecture, REST APIs, and secure authentication. Considering the user's needs is important because software that does not solve the client's actual problems will not be successful, even if it is technically well designed.

## How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?

I approached software design by first identifying the client's requirements and then selecting technologies and architectural patterns that best supported those requirements. Throughout this course, I learned the importance of planning before coding. In the future, I would continue using UML diagrams, software design documents, and architectural analysis to evaluate multiple solutions before beginning implementation. I would also spend more time considering scalability, security, and long-term maintenance early in the design process to create software that is easier to expand and support.
